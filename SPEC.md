# LemmaScript — Implementation Specification

**Version:** 0.1
**Date:** March 2026
**Status:** Specification for rewrite, informed by Phase 0 findings

---

## 1. Overview

LemmaScript is a verification toolchain for TypeScript. The user writes TypeScript with `//@ ` annotations. Ghost definitions and proofs live in `.spec.lean` files (plain Lean 4). The toolchain generates a Lean embedding and Lean checks it.

The toolchain has three components:
1. **`lsc` CLI** (Node.js) — parses TS, generates Lean
2. **LemmaScript Lean library** — re-exports Velvet/Loom, provides any needed macros
3. **Pre-built oleans** — so user projects don't recompile the dependency chain

---

## 2. User Project Structure

LemmaScript is a library. Each user project depends on it.

```
my-app/
  src/
    withdraw.ts              ← TypeScript source
    withdraw.spec.lean       ← ghost definitions, lemmas, proofs
    binarySearch.ts
    binarySearch.spec.lean
  lean/
    lakefile.lean            ← requires LemmaScript
    lean-toolchain
    MyApp/
      Withdraw.lean          ← generated by lsc (gitignored)
      BinarySearch.lean      ← generated by lsc (gitignored)
  package.json              ← depends on @lemmascript/tools
  .gitignore                ← includes lean/MyApp/
```

**Conventions:**
- TS source and spec files live together in `src/`.
- Generated Lean lives in `lean/<ProjectName>/`, never edited by users.
- The spec file for `foo.ts` is `foo.spec.lean`. If it doesn't exist, no spec import is generated.
- `lsc` derives the Lean module path from the file name: `src/binarySearch.ts` → `lean/MyApp/BinarySearch.lean`.

**`lsc init`** scaffolds the `lean/` directory with `lakefile.lean`, `lean-toolchain`, and the LemmaScript dependency.

## 3. The `//@ ` Annotation Language

### 3.1 Annotation Kinds

Annotations are TypeScript comments of the form `//@ <keyword> <expression>`.

| Keyword | Placement | Meaning |
|---------|-----------|---------|
| `requires` | Before first statement of function body | Precondition |
| `ensures` | Before first statement of function body | Postcondition (`result` refers to return value) |
| `invariant` | Before first statement of loop body | Loop invariant |
| `decreases` | Before first statement of loop body | Termination metric |
| `type` | Before first statement of function body | Type override for a variable (see §3.3) |

### 3.2 Spec Expression Grammar

The expression language is a subset of TypeScript with verification extensions.

```
expr     := implies
implies  := or ('==>' implies)?          // right-associative
or       := and ('||' and)*
and      := compare ('&&' compare)*
compare  := add (cmpOp add)?
cmpOp    := '===' | '!==' | '>=' | '<=' | '>' | '<'
add      := mul (('+' | '-') mul)*
mul      := unary (('*' | '/' | '%') unary)*
unary    := '!' unary | '-' unary | postfix
postfix  := atom ('.' ident | '[' expr ']' | '(' args ')')*
atom     := NUMBER | IDENT | 'true' | 'false' | 'result'
          | 'forall' '(' IDENT (':' TYPE)? ',' expr ')'
          | 'exists' '(' IDENT (':' TYPE)? ',' expr ')'
          | '(' expr ')'
TYPE     := 'nat' | 'int'
```

**`result`** in ensures refers to the function's return value. In loop invariants when the function has an early return inside the loop, `result` refers to the tentative return value (see §5.3).

**`forall(k, P)`** quantifies `k` as `Int` by default. **`forall(k: nat, P)`** quantifies as `Nat`.

### 3.3 Type Annotations

By default, all TS `number` variables map to `Int` in Lean. The `type` annotation overrides this:

```
//@ type <varname> nat
```

This causes the variable to be declared as `Nat` in the generated Lean. Effects:
- `let mut i : Nat := 0` instead of `Int`
- `arr[i]!` instead of `arr[i.toNat]!`
- `arr.size - i` is Nat subtraction (for `decreases` clauses)
- Ghost function calls pass the variable directly (ghost functions should take `Nat` if the caller uses `Nat`)

Use `nat` for non-negative loop counters and array indices. Use `Int` (default) for values that can be negative (e.g., `hi` in binary search).

## 4. Spec Expression → Lean Translation

The translation is purely syntactic. The codegen does not infer types beyond what `//@ type` annotations provide.

### 4.1 Operator Mapping

| Spec | Lean |
|------|------|
| `===` | `=` |
| `!==` | `≠` |
| `>=` | `≥` |
| `<=` | `≤` |
| `>` | `>` |
| `<` | `<` |
| `&&` | `∧` |
| `\|\|` | `∨` |
| `!` | `¬` |
| `==>` | `→` |
| `+`, `-`, `*`, `/`, `%` | `+`, `-`, `*`, `/`, `%` |

No normalization of `≥` to `≤` or similar. Lean and `loom_solve` handle all comparison directions.

### 4.2 Special Forms

| Spec | Lean | Notes |
|------|------|-------|
| `arr.length` | `arr.size` | Always `Nat`. Lean coerces to `Int` in mixed contexts. |
| `arr[e]` where `e` is Nat-typed | `arr[e]!` | Direct Nat index. |
| `arr[e]` where `e` is Int-typed | `arr[e.toNat]!` | Int-to-Nat conversion for indexing. |
| `f(a, b)` | `f a b` | Ghost function call (Lean application syntax). |
| `Math.floor(e)` | `e` | Int division in Lean is already floor. |
| `result` | `res` in ensures; `result` in invariants with return-in-loop | See §5.3. |
| `forall(k, P)` | `∀ k : Int, P'` | Default Int quantification. |
| `forall(k: nat, P)` | `∀ k : Nat, P'` | Nat quantification. |

### 4.3 Nat-Typing Rules

An expression is Nat-typed if:
- It's a variable declared with `//@ type <v> nat`
- It's a quantified variable with `: nat` in the quantifier
- It's `arr.length` (i.e., `arr.size`)
- It's an arithmetic expression (`+`, `-`, `*`, `/`, `%`) where both operands are Nat-typed
- It's a non-negative numeric literal

The Nat-typing determines:
- Whether `.toNat` is needed for array indexing
- Whether `decreases` needs `.toNat` wrapping

### 4.4 Implication Flattening

`(A && B) ==> C` is emitted as `A → B → C` (curried). This is standard Lean idiom and `loom_solve` handles it better than `(A ∧ B) → C`.

### 4.5 Conjunction Splitting

Top-level `&&` in `requires`, `ensures`, and `invariant` annotations is split into separate clauses:

```
//@ ensures result >= -1 && result < arr.length
```

generates:

```lean
ensures res ≥ -1
ensures res < arr.size
```

## 5. Statement Translation (TS → Lean)

### 5.1 Basic Statements

| TypeScript | Lean | Notes |
|-----------|------|-------|
| `let x = e` | `let mut x : T := e'` | Mutable. `T` from type map (§3.3). |
| `const x = e` | `let x := e'` | Immutable. No type annotation. |
| `x = e` | `x := e'` | Assignment. |
| `return e` | `return e'` | Outside loops only (see §5.3). |
| `if (c) { ... }` | `if c' then ...` | |
| `if (c) { ... } else { ... }` | `if c' then ... else ...` | |
| `if (c) { ... } else if ...` | `if c' then ... else if ...` | Chained. |
| `while (c) { ... }` | `while c' invariant ... decreasing ... do ...` | See §5.2. |

All expressions `e` in the table above are translated using the spec expression rules (§4).

### 5.2 While Loops

```typescript
while (condition) {
  //@ invariant P
  //@ invariant Q
  //@ decreases D
  body
}
```

generates:

```lean
while condition'
  invariant P'
  invariant Q'
  decreasing D'          -- .toNat wrapped if D is Int-typed
do
  body'
```

**Decreasing clause:** If the decreases expression is Nat-typed (§4.3), emit directly. Otherwise, wrap in `.toNat`.

### 5.3 Return Inside Loops

TypeScript allows `return` inside `while` loops. Velvet does not. The codegen transforms this pattern:

**Input:**
```typescript
function f(...): number {
  ...
  while (condition) {
    ...
    if (guard) { return value; }
    ...
  }
  return default;
}
```

**Transformation:**
1. Introduce `let mut result : T := default'` (from the `return default` after the loop).
2. Replace `return value` inside the loop with `result := value'; break`.
3. Add `done_with result ≠ default' ∨ ¬(condition')` to the while.
4. Replace the final `return default` with `return result`.

**Placement:** The `result` variable is declared after the user's leading `let` declarations but before the `while` loop. This ensures the Velvet state tuple has `result` in the right position.

**`result` in annotations:** The user may reference `result` in `//@ invariant` annotations inside the loop to state properties about the tentative return value. This is necessary when `loom_solve` cannot infer the needed invariant. If `loom_solve` fails, the LLM proof filler (lean-lsp-mcp) can propose the missing invariant.

**Open question:** Whether `result` should be user-visible in annotations or whether this should be fully automated (by the LLM, not the codegen). Current status: user writes it. This may change.

## 6. Generated Lean File Structure

For a TS file `src/foo.ts`, `lsc` generates `lean/<Project>/Foo.lean`:

```lean
/-
  Generated by lsc from foo.ts
  Do not edit — re-run `lsc check` to regenerate.
-/
import Velvet.Syntax
import Velvet.Std
import <Project>.FooSpec        -- only if src/foo.spec.lean exists

set_option loom.semantics.termination "total"
set_option loom.semantics.choice "demonic"

method foo (params...) return (res : RetType)
  require ...
  ensures ...
  do
    ...

prove_correct foo by
  loom_solve
```

**Spec file import:** The codegen checks for `src/foo.spec.lean`. If it exists, it generates the import. The spec file is copied (or symlinked) into the `lean/` directory so Lake can find it. The Lean module name is derived by convention: `foo.spec.lean` → `<Project>.FooSpec`.

### 6.1 When `loom_solve` Fails

The generated file always uses `prove_correct foo by loom_solve`. If this fails:

1. `lsc check` reports the unsolved goals with source-mapped line numbers.
2. The user (or LLM) adds lemmas to the `.spec.lean` file.
3. For goals that need additional invariants or proof steps, the user can add a `foo.proof.lean` file that provides an alternative proof tactic. (Exact mechanism TBD.)

## 7. Type Mapping

| TypeScript | Lean (default) | With `//@ type v nat` |
|-----------|----------------|----------------------|
| `number` | `Int` | `Nat` |
| `number[]` | `Array Int` | — |
| `boolean` | `Bool` | — |
| `string` | `String` | — |

Function parameters are always mapped by their TS type. The `//@ type` annotation only applies to local variables.

Array element type is always `Int` (TS `number[]` → `Array Int`). This may be extended later.

## 8. `lsc` CLI

```
lsc init                    — scaffold lean/ directory in a user project
lsc check <file.ts>        — extract + generate + lake build
lsc check --watch           — re-check on file changes
lsc gen <file.ts>           — extract + generate only (print Lean to stdout)
lsc extract <file.ts>       — extract only (print IR JSON to stdout)
```

**`lsc check` workflow:**
1. Parse TS file with ts-morph → IR.
2. Generate Lean source from IR.
3. Copy/symlink spec file into `lean/` if it exists.
4. Write generated Lean to `lean/<Project>/<Name>.lean`.
5. Run `lake build <Project>.<Name>` (or invoke `lean` directly with pre-built `LEAN_PATH`).
6. Report results. Map Lean error locations back to TS source lines.

**`lsc init` workflow:**
1. Create `lean/` directory with `lakefile.lean`, `lean-toolchain`.
2. `lakefile.lean` requires LemmaScript.
3. Download pre-built oleans for LemmaScript + dependencies.

## 9. LemmaScript Lean Library

The LemmaScript Lean library (`LemmaScript` package) provides:

1. **Re-exports of Velvet and Loom.** Users import `LemmaScript` and get everything.
2. **Any LemmaScript-specific macros.** Currently none — Velvet's `method`, `prove_correct`, `loom_solve` are used directly. Macros may be added if the generated Lean needs constructs Velvet doesn't provide.

The library depends on:
- Velvet (which depends on Loom, which depends on mathlib)
- Z3 and cvc5 (downloaded by the lakefile)

Pre-built oleans are distributed so user projects skip compilation.

## 10. Findings from Phase 0

These are empirical constraints discovered during prototyping. They inform the design but should be re-validated during implementation.

### 10.1 Int vs Nat

- All TS `number` variables default to `Int` in Lean.
- Variables annotated `//@ type v nat` become `Nat`.
- Mixing Int and Nat in invariants/postconditions is fine — Lean coerces. But quantifier types must be consistent: don't quantify `k : Int` in an invariant and `k : Nat` in the ensures for the same property.
- `arr.size` is `Nat`. Lean coerces to `Int` automatically in comparisons with `Int` values. No explicit `↑` needed.
- Array indexing with `Nat` uses `arr[i]!`. With `Int` uses `arr[i.toNat]!`.
- Recursive ghost functions on array indices should take `Nat` (natural for structural recursion). The calling code should use `//@ type` to make the index variable `Nat`.
- When a ghost function takes `Nat` but the caller has an `Int`, a bridge lemma may be needed in the `.spec.lean` (e.g., connecting `(i + 1).toNat` to `i.toNat + 1`). This is a known friction point.

### 10.2 `loom_solve` Capabilities

- Discharges most verification conditions for array algorithms automatically (binary search, linear search, array sum).
- Handles `≥`, `>`, `≤`, `<` — no normalization needed.
- Handles quantifiers, array access, sorted predicates.
- Needs ghost functions tagged `@[grind, loomAbstractionSimp]` to reason about them.
- Recursive ghost functions may need explicit step lemmas (e.g., `sumTo_step`) tagged with the same attributes.
- Cannot invent loop invariants. If the invariant is missing or too weak, it fails with an unsolved goal.
- Cannot bridge `(i + 1).toNat` to `i.toNat + 1` without a lemma.

### 10.3 Velvet Specifics

- `method` syntax handles `require`, `ensures`, `invariant`, `done_with`, `decreasing`, `break`, mutable variables.
- `return` is not supported inside loops — must use `break` + result variable.
- `done_with` captures what is true when the loop exits (by condition or by break).
- Variable declaration order determines the state tuple structure. This affects proof state variable names but not provability.
- `prove_correct` + `loom_solve` is the standard proof strategy.
- `loom_solve!` shows unsolved goals for debugging.

### 10.4 Decreasing Clauses

- For Nat-typed expressions: emit directly (e.g., `decreasing arr.size - i`).
- For Int-typed expressions: wrap in `.toNat` (e.g., `decreasing (hi - lo + 1).toNat`).
- Avoid mixing: `(arr.size - i).toNat` where `arr.size : Nat` and `i : Int` causes `loom_solve` failures. Use `arr.size - i.toNat` or make `i` a `Nat`.

## 11. What This Spec Does Not Cover (Future Work)

- **For-of loops** (`for (const x of arr)`)
- **Object/record types** (`{ balance: number }`)
- **Higher-order functions** (`arr.map`, `arr.filter`)
- **Async/await**
- **Multiple functions per file** with inter-function calls
- **Array mutation** (`arr[i] = v` as opposed to functional update)
- **Error reporting** (mapping Lean errors back to TS source locations)
- **VS Code extension**
- **LLM proof filling integration**
- **Pre-built olean distribution**

## 12. Implementation Order

1. **Spec expression parser** — tokenizer, recursive descent parser, AST. Tested independently with unit tests.
2. **Lean emitter** — AST → Lean string, using EmitContext for type information. Tested independently.
3. **TS extractor** — ts-morph, produces IR. Tested on example files.
4. **Code generator** — IR → Lean source. Uses parser + emitter. Tested by diffing against known-good Lean output.
5. **`lsc` CLI** — wires extractor → codegen → Lake. Integration tested on the three examples.
6. **Lean library** — minimal, re-exports Velvet/Loom. Verified by building the examples.
7. **Project scaffolding** (`lsc init`) — generates `lean/` directory structure.

Each step has clear inputs and outputs. Test at each boundary.
